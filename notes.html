Ooh I think I’ve found a solutions to the big problem I’ve been working on. The problem is pretty simple in essence “Which program representation do we give meta-level optimizers access to?”.

Doing optimizations on the surface language is a big no-no - the surface language contains too many user-facing details that optimizers shouldn’t have to care about. Well alright, what about the core language? Problem with that is that the core language doesn’t give enough control over performance. You can’t fuse allocations or change data representations or transform recursive functions into loops.

Well alright, what about STG? That kinda works. But now we have another problem - well, not really a problem, but a wart. Userspace optimizers sit on top of the “default” way to lower core to STG. The built-in lowering - the lowering procedure that’s a part of the compiler - doesn’t do any smart optimizations. “Well, why not just make the STG lowering pass smarter? Make it optimize better?”. But isn’t complex built-in optimizers exactly what we were trying to avoid in the first place? The meta-level is supposed to have complete control over how optimization works, but we’re forcing it to share space with the “default” optimizer
So basically, my solution to the problem is to get rid of the built-in lowering pass entirely. Metaprograms will have complete control over compilation to STG - none of that will be implemented in the compiler at all

.. guh, codegen is delayed then? Seems like it
I’ll have to implement more of the metalanguage. But maybe that’s a good thing - forcing me to do this. I actually haven’t implemented any metaprogramming capabilities - I’ll be forced to focus on it, which I’d say is a good thing

To be clear, there’s nothing special about STG here - the important thing is that we give metaprograms access to a low-level program representation

Γ ⊢ A : Type   Γ, x : A ⊢ B : Prop
———————————————————————————————————
       Γ ⊢ <x : A, B> : Type

Where Type is the universe of object level types and Prop the universe of meta level types. So really just a sigma type where the second component is erased. Anyway, we could define a proposition

Linear : [Γ ⊢ a → b] → Prop

Which demonstrates that a function is linear. We then have

LinearFunction : Π (a b : Type) → <f : a → b, Linear [. f]>

Then add some nice syntax

a ⊸ b = LinearFunction a b

And boom, linear types. I’m reasonably sure this sort of thing has been studied already, just need to find the literature. Possibly related topics: 1. Church vs Curry style typing 2. ATS’s viewtypes

A few weeks ago I started on a completely new version of my language - not only the compiler has changed, but the language itself has. It boils down to a change in philosophy, from “Let’s restrict the language so we’re guaranteed to always have good performance” to “Let’s not restrict the language at all and recover performance where we can”.

Other: What led you to add logic programming aspects to your language?

Me: Short answer: Logic programming seems particularly suited to writing program optimizers. Thus, the metaprogramming level of my language is LP Long answer: My language is really two languages in one - one which runs only at compile time, and one which runs at runtime. I want advanced features like dependent types, algebraic effects, etc, but I also didn’t want them to have a significant negative impact on performance. So I put all of them in the comptime language - they could only be used when they could be eliminated fully at comptime. More generally: High abstractions could be added, but they had to be fully static

The problem with that is that restricting those kinds of features purely to compile time has a significant impact on expressivity - and expressivity is the entire point of having those high-level features in the first place. So now I’m redesigning my language around a different idea: We let the runtime language be as high-level as it can be, but the compile time language is specifically designed for writing optimizers The cool thing about this is that since optimization is put in userspace, your optimizers can be as domain-specific as you need! Optimizers for specific patterns can be written easily. This is in contrast to compilers, which can only implement language-wide optimizations - and true domain-specific optimizations take more effort to create Back to LP 😅 LP seems quite good for writing optimizers, so the compile time half of my language with be LP

So yeah, logic programming! The meta level will have LP features along the lines of λProlog. The object level will have full dependent types, purity, etc

Optimizing the Reader monad. Uses contextual types. Note that optimize and runReader are very similar

Reader e r = e ~> r

(>>=) : Reader e a ~> (a ~> Reader e b) ~> Reader e b
act >>= k = \e. k (act e)

pure : a ~> Reader e a
pure x = \_. x

optimize : [⊢ e] -> [⊢ Reader e a] -> [⊢ a]
optimize _ [. pure ~x] = x
-- forall a b.
-- act0 : [⊢ Reader e a]
-- act1 : [a ⊢ Reader e b]
optimize e [~act0 >>= \r. ~act1] =
  let r = optimize e act0
  in optimize e (act1 with r)
optimize e act = [. ~act ~e]

Evaluation

ask = id

optimize 3 (ask >>= \x. pure (x + 2))
==> let r = optimize 3 [. ask]
    in optimize 3 ([x. pure (x + 2)] with r)
==> let r = [. ask 3]
    in optimize 3 ([x. pure (x + 2)] with r)
==> optimize 3 ([x. pure (x + 2)] with [. ask 3])
==> optimize 3 ([. pure (ask 3 + 2)])
==> [. ask 3 + 2]

Almost all the function calls removed! If we wanted to, we could add a case to optimize to turn ask 3 into 3, but we won’t do that for now. However, this approach has an issue. Let’s define >> and attempt to optimize it

act0 >> act1 = act0 >>= \_. act1
optimize 3 (ask >> pure (3 + 2))
==> [. (ask >> pure (3 + 2)) 3]

Oh no! This one got stuck. >> doesn’t match optimize’s >>= or pure cases, so it goes to the last case. The problem is thus:

We would like optimize to recognize that act0 >> act1 and act0 >>= \_. act1 are equivalent. We could add a case for >> to optimize, but that breaks abstraction. You have to change the library every time you want to optimize a new operator in client code.

First solution: Open pattern matching. The library defines a signature for optimize, and clients can add whatever cases they want. This has almost the same problem as before though! >> and >>= can be optimized in the same way, but we have to write new code for each of them! This is not really a solution. Let’s refine the problem statement:

We would like optimize to recognize that act0 >> act1 and act0 >>= \_. act1 are equivalent. We could add a case for >> to optimize, but we shouldn’t have to! >> is defined completely in terms of >>=, so optimize should be able to optimize it without any changes to itself.

This might involve adding reduction rules to dynamic fragments, it might involve theorem proving, I’m not sure. But I think I’m very clear on what the problem is at the moment.

There’s a tension I’m feeling with regard to 2LTT. How does one decide how high- or low-level the meta language and object language should be? Let’s take things to the extreme first. The object language will be something close to the abstraction level of assembly, while the meta language is at the level of Agda. In order to program effectively in this language, every program has to be a compiler. Even a feature as simple as functions would be implemented in userspace. I realize that this is what I (and probably a lot of others) have dreamt about - a language where literally everything is implemented in userspace. You don’t have to implement everything from scratch though, of course you can write compilers in terms of existing compilers.

For some reason though, this feels uncomfortable. Now that the “most powerful language” is theoretically achievable, it doesn’t seem like a good idea. Here’s why: Being able to program at the lowest level doesn’t mean your programs are efficient. Take this example: Idris can compile to C. C is a fast language. So why is Idris not as fast as C? Because the compiled C code uses a lot of abstractions.

Key point: Efficiency does not depend on the low-levelness of a language. Rather, it depends on the lack of abstractions.

In Idris’s case, the abstractions are written using C, so they’re pretty fast. But the high amount of abstractions means the code is still high level.

Key point: You can write high-level code in a low-level language.

This is the problem. With this “programs are compilers” setup, you still have to drop down to the lower levels of abstraction when you want real speed. Because we’re not optimizing anything - we’re not eliminating any abstractions.

Oh! That’s it! The real flaw is that we’re missing something - optimization. This approach makes it easy to generate code. However, to have real efficiency we also need to make it easy to optimize code - to remove abstractions. That’s what I need to focus on.

Researching query driven compilation at the moment so I can implement it. The idea is that every operation computes everything from scratch. So the operation to get the type of a toplevel definition would 1. Parse a module 2. Typecheck the whole module 3. Get the type of the definition. To make this performant, you make every compiler optimization memoized, so each query is always performed at most once. It abstracts over all the disparate incremental features that compilers have, allowing you to implement them all using one system, and implement them much more easily

The Free type forms a monad, but the >>= implementation doesn’t do any actual computations - it just builds up a bigger Free value. Free is the simplest monad.

data Free f a
  = Pure a
  | Roll (f (Roll f a))
instance Functor f => Functor (Free f) where
  fmap g (Roll fx) = Roll (fmap (fmap g) fx)
  fmap g (Pure x)  = Pure (g x)
instance Functor f => Monad (Free f) where
  pure = Pure
  Pure x >>= g = g x
  Roll fx >>= g = Roll (fmap (>>= g) fx)

The Free type is a framework for DSLs. You build up programs with >>=, then define interpreter functions. Question: How does the free monad allow for composition of separate DSLs? E.g algebraic effects allow DSLs to be composed arbitrarily, thanks to them being based on free monads.
